\begin{abstract}
We demonstrate LLARS, an open-source platform that bridges the gap between domain experts and developers for systematic evaluation of LLM outputs.
The platform provides three integrated modules that can be used together or independently:
\textbf{Collaborative Prompt Engineering} lets domain experts and developers co-author prompts in a shared real-time workspace with version control and instant LLM testing.
\textbf{Batch Generation} systematically produces outputs for all combinations of prompts, LLMs, and data items with built-in cost tracking.
\textbf{Large-Scale Evaluation} supports human and LLM evaluators with live agreement metrics---applicable to outputs from any source.
When combined, these modules form a complete pipeline from prompt development through rigorous assessment.

Organizations increasingly build LLM-based systems for sensitive domains such as psychosocial counseling, legal analysis, and medical documentation.
Developing these systems requires writing effective prompts, selecting suitable models, and rigorously evaluating outputs---tasks that demand both domain expertise and technical skill.
Domain experts understand \emph{what} constitutes quality in their field, while developers know \emph{how} to structure prompts and configure LLM pipelines.
Yet these two groups typically work in isolation: prompts are iterated in shared documents without version control, model outputs are evaluated ad hoc, and no single tool supports the full cycle from collaborative prompt development through systematic evaluation.

We present LLARS (LLM Assisted Research System), an open-source platform that bridges this gap with three integrated modules (Figure~\ref{fig:pipeline}).
\emph{Collaborative Prompt Engineering} lets domain experts and developers co-author prompts in a real-time CRDT-based editor with version control and instant LLM testing---so both groups iterate in one workspace rather than exchanging documents.
\emph{Batch Generation} then systematically produces outputs for all combinations of prompts, models, and data items with cost tracking, enabling rigorous cross-model comparison.
\emph{Evaluation} lets human evaluators and LLM judges assess these outputs side by side---via rating, ranking, labeling, or pairwise comparison---while live inter-rater agreement metrics (Krippendorff's~$\alpha$, Cohen's~$\kappa$, and others) reveal where human and machine judgment converge or diverge.
Each module also operates independently with JSON/CSV export, and the platform connects to any OpenAI-compatible LLM provider including locally hosted models via Ollama.

Table~\ref{tab:comparison} surveys twelve existing platforms along these dimensions.
Agenta~\cite{agenta2024} combines prompt management with LLM-as-judge scoring but does not offer real-time co-editing---essential when domain experts and developers jointly refine prompts---and lacks structured human evaluation campaigns with agreement metrics.
The observability platforms Phoenix~\cite{phoenix2024}, Langfuse~\cite{langfuse2024}, and W\&B~Weave~\cite{weave2024} provide dataset experiments and per-output feedback but focus on tracing and debugging rather than systematic batch generation or structured multi-evaluator campaigns.
ChainForge~\cite{arawjo2024chainforge} supports visual multi-prompt comparison across models but does not include human evaluation and relies on custom metric functions instead of dedicated LLM judge prompts.
The annotation platforms Label~Studio~\cite{labelstudio2024} and Argilla~\cite{argilla2024} offer structured human assessment workflows---Label~Studio also computes basic annotator agreement---but both require outputs to be generated externally, disconnecting evaluation from the prompt development process.
The commercial platforms LangSmith~\cite{langsmith2024}, Braintrust~\cite{braintrust2024}, Maxim~\cite{maxim2024}, and Vellum~\cite{vellum2024} link prompt editing to evaluation with strong LLM-as-judge capabilities but are closed-source, lack collaborative editing, and do not include batch generation as an explicit pipeline stage.

By unifying collaborative prompt engineering, Cartesian-product batch generation, and hybrid human--LLM evaluation with comprehensive agreement analytics, LLARS enables a workflow where developers handle LLM integration, prompt versioning, and batch configuration while domain experts contribute quality criteria and evaluation judgments---both working in the same shared environment from prompt development through rigorous assessment.


\section{Platform Overview}

LLARS is a containerized web application built with Vue.js (frontend), Flask (backend), and MariaDB (storage).
Real-time collaboration is provided by a dedicated Yjs~\cite{jahns2023yjs} server, and LLM access follows the OpenAI-compatible API structure, allowing teams to connect commercial providers (OpenAI, Anthropic, Google) as well as locally hosted models via Ollama.
All interactions---prompt editing, generation progress, and evaluation updates---are synchronized via WebSocket.

\subsection{Collaborative Prompt Engineering}

Developing effective prompts for domain-specific LLM systems requires iterative refinement by those who understand the domain and those who understand prompt design.
The prompt engineering module (Figure~\ref{fig:prompt-editor}) addresses this through a real-time collaborative editor built on Yjs~\cite{jahns2023yjs}, a CRDT framework that synchronizes edits at character granularity, merging concurrent changes without conflicts.

Users compose prompts as reorderable message blocks (e.g., system instructions, user templates) and define template variables using \texttt{\{\{variable\}\}} syntax.
Variables are automatically extracted and displayed in a shared palette; during testing and batch generation, they are substituted with domain-specific data.
Any collaborator can test the current prompt independently: clicking \emph{Test} captures an atomic snapshot, fills variables with sample data, sends the prompt to a user-selected LLM, and streams the response---without interrupting concurrent editing.
Multiple users can test against different models or data in parallel, enabling rapid exploration of prompt behavior across configurations.

A git-inspired version control system records each commit with multi-author attribution, enabling version comparison and rollback.
Finalized prompts are exported as JSON or passed directly to batch generation.

\subsection{Batch Generation}

With prompts refined, the batch generation module scales output creation across combinations of data, prompts, and models (Figure~\ref{fig:batch-generation}).
A guided wizard walks users through source selection, prompt and model configuration, and parameter tuning including budget limits.
Sources can be existing evaluation scenarios, uploaded CSV or JSON files, or---for self-contained prompts that require no input data, such as creative text generation or instruction-following benchmarks---no source at all.
The system generates the full Cartesian product: $N$~data items $\times$ $M$~prompts $\times$ $K$~models.
Template variables in prompts are automatically matched to data columns by name---a CSV column \texttt{content} fills every \texttt{\{\{content\}\}} placeholder across all selected prompts.

Users can register custom LLM providers with their own API keys; these models then appear throughout the platform---in prompt testing, batch generation, and as LLM evaluators.
Before execution, LLARS estimates costs based on model pricing and token projections; optional budget caps automatically pause jobs that exceed thresholds.
All generation progress---including token-level output streaming---is broadcast via WebSocket, allowing team members to join and monitor running jobs in real time.
Each completed output retains full provenance---source item, prompt template, model, token count, and cost---exportable as CSV or JSON.
Completed batches can be transferred directly into the evaluation module through the integrated Scenario Wizard, preserving model attribution for systematic cross-model comparison.

\subsection{Evaluation}

The evaluation module enables human evaluators and LLM evaluators to assess outputs within a unified framework, supporting four general evaluation types (Figure~\ref{fig:scenario-wizard}): \emph{rating} with configurable multi-dimensional Likert scales and weighted dimensions (e.g., coherence, fluency, relevance), \emph{ranking} where evaluators sort items into ordered buckets, \emph{labeling} for categorical classification (binary or multi-class), and pairwise \emph{comparison}.
The framework is extensible with domain-specific types; LLARS includes two for psychosocial counseling research (Section~\ref{sec:counseling}).

An AI-assisted Scenario Wizard guides scenario creation through five steps---data upload, task selection, configuration, team assembly, and review.
The wizard analyzes uploaded data structures using deterministic pattern matching with an LLM fallback for ambiguous cases, and recommends the most suitable evaluation type along with matching presets.
Scenarios can also be created directly from batch generation outputs, preserving model attribution so that evaluators compare outputs across LLMs for the same input.

Scenario owners control how items are distributed: all items to every evaluator for full inter-rater analysis, or configurable subsets for faster throughput.
Both human evaluators and LLM evaluators participate in the same scenarios; LLM evaluators use type-specific prompts that administrators can customize per scenario type.
As evaluations are submitted, LLARS computes agreement metrics in real time---Krippendorff's~$\alpha$, Cohen's~$\kappa$, Fleiss'~$\kappa$, Kendall's~$\tau$ and~$W$, Spearman's~$\rho$, and intraclass correlation~(ICC), among others---and scenario owners can filter by evaluator type (human--human, LLM--LLM, or human--LLM) to analyze where machine and human judgment diverge.
All evaluation data, metrics, and configurations can be exported as JSON or CSV for further analysis.

\section{Application: Online Counselling}
\label{sec:counseling}

LLARS is deployed in ongoing research on CAIA~\cite{steigerwald2025caia}, an AI-assisted support system for text-based psychosocial counselling with seven LLM-driven functionalities---from case summarisation to hypothesis generation.
Counselling researchers and developers used the collaborative prompt editor to jointly develop the prompts underlying these functionalities, testing iteratively against counselling data until outputs met professional standards.
For one of these functionalities---automated subject line generation~\cite{steigerwald2025subjectline}---the complete pipeline was exercised: batch generation produced 253~subject lines across 11~LLMs at different quantisation levels, and five counselling professionals together with an AI evaluator rated the outputs, yielding 1,518~assessments.
LLARS's cross-model comparison directly informed the selection of a privacy-preserving open-source model for local deployment.

These projects also motivated two domain-specific evaluation types (Figure~\ref{fig:scenario-wizard}):
\emph{Mail Rating} for assessing AI-generated counsellor responses on dimensions such as coherence and quality, and \emph{Authenticity} for classifying counselling threads as real or AI-generated, with ground truth labels enabling classification accuracy alongside inter-rater agreement.
