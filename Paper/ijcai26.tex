\begin{abstract}
We demonstrate LLARS, an open-source platform that bridges the gap between domain experts and developers for building LLM-based assistants.
LLARS integrates three modules into an end-to-end pipeline:
\textbf{Collaborative Prompt Engineering} for real-time co-authoring with version control and instant LLM testing,
\textbf{Batch Generation} for systematic output production across prompts\,$\times$\,models\,$\times$\,data with cost control, and
\textbf{Evaluation} for human and LLM assessors to rate, rank, label, or compare outputs with live inter-rater agreement metrics.
Interviews with six domain experts and three developers in psychosocial counselling confirmed that LLARS eliminates the ``translation work'' between disciplines and enables domain experts to directly co-author prompts and conduct evaluations independently.

Organizations building LLM-based assistants for sensitive domains~\cite{chen2024critical,ling2025domain}---psychosocial counseling~\cite{stade2024behavioral}, legal analysis~\cite{katz2024bar}, medical documentation~\cite{vanveen2024clinical}---must write effective prompts, select suitable models, and rigorously evaluate outputs, requiring both domain expertise (understanding \emph{what} constitutes quality) and technical skill (structuring prompts and configuring LLM pipelines)~\cite{zamfirescu2023johnny,schulhoff2024prompt}.
Yet these two groups typically work in isolation: prompts are iterated in shared documents without version control, and no single tool supports the full cycle from collaborative prompt development through systematic evaluation.

We present LLARS (LLM Assisted Research System), an open-source co-designed platform that closes this gap with three integrated modules (Figure~\ref{fig:pipeline}).
\emph{Collaborative Prompt Engineering} lets domain experts and developers co-author prompts in a shared real-time editor with version control and instant LLM testing.
\emph{Batch Generation} systematically produces the Cartesian product of prompts\,$\times$\,models\,$\times$\,data items with cost estimation and budget control.
\emph{Evaluation} lets human and LLM evaluators assess outputs side by side---via rating, ranking, labeling, or pairwise comparison---with live agreement metrics (Krippendorff's~$\alpha$, Cohen's~$\kappa$, and others).
Each module can be used independently with JSON/CSV export, and the platform connects to any OpenAI-compatible provider including locally hosted models.

Table~\ref{tab:comparison} surveys twelve existing platforms.
Tracing-oriented tools (Phoenix~\cite{phoenix2024}, Langfuse~\cite{langfuse2024}, W\&B~Weave~\cite{weave2024}) and prompt managers (Agenta~\cite{agenta2024}, ChainForge~\cite{arawjo2024chainforge}) lack structured human evaluation with agreement metrics.
Annotation platforms (Label~Studio~\cite{labelstudio2024}, Argilla~\cite{argilla2024}) disconnect evaluation from prompt development.
Closed-source tools (LangSmith~\cite{langsmith2024}, Braintrust~\cite{braintrust2024}, Maxim~\cite{maxim2024}, Vellum~\cite{vellum2024}) omit collaborative editing and batch generation.
LLARS is the first open-source platform to unify all three stages with agreement analytics.


\section{Platform Overview}

LLARS runs as a containerized web application connecting to any OpenAI-compatible LLM provider, including commercial services and locally hosted models via Ollama.
Teams register providers with their own API keys; models then appear throughout the platform.

\subsection{Collaborative Prompt Engineering}

The prompt engineering module (Figure~\ref{fig:prompt-editor}) lets multiple users edit the same prompt simultaneously in a real-time collaborative editor built on Yjs~\cite{jahns2023yjs}.
Prompts are structured as reorderable message blocks---system instructions, few-shot examples, user templates---with \texttt{\{\{variable\}\}} placeholders that the platform extracts into a shared palette for the team.
Variables keep domain data separate from prompt logic: users assign sample data to each variable and can swap between inputs (e.g.\ two counselling threads) to verify generalisation instead of overfitting to a single example.
When the prompt is ready, variables map directly to data columns in batch generation, so the same template processes an entire dataset without modification.
Any collaborator can test the current prompt against a selected LLM while editing continues, and a version control system tracks every revision with multi-author attribution.

\subsection{Batch Generation}

The batch generation module (Figure~\ref{fig:batch-generation}) produces outputs at scale.
A wizard walks users through selecting a data source (scenario, CSV/JSON upload, or none), choosing prompts and models, and setting budget limits.
LLARS generates the full Cartesian product---$N$~data items $\times$ $M$~prompts $\times$ $K$~models---matching template variables to data columns automatically, with cost estimation and optional budget caps.
Every result records its full provenance and is exportable as CSV or JSON.
Completed batches transfer directly into evaluation, preserving model attribution for cross-model comparison.

\subsection{Evaluation}

The evaluation module brings human and LLM evaluators together in a single framework.
Four evaluation types are available (Figure~\ref{fig:scenario-wizard}): \emph{rating} with multi-dimensional Likert scales, \emph{ranking} into ordered buckets, \emph{labeling} for classification, and pairwise \emph{comparison}; the type system is extensible (Section~\ref{sec:counseling}).
An AI-assisted Scenario Wizard analyzes uploaded data to recommend the most suitable type; scenarios can also be created from batch generation outputs.

Where most tools limit LLMs to pairwise judging~\cite{zheng2024judging}, LLARS treats them as full evaluation participants alongside humans.
Agreement metrics update live---Krippendorff's~$\alpha$, Cohen's and Fleiss'~$\kappa$, Kendall's~$\tau$, and ICC---filterable by evaluator pairing (human--human, LLM--LLM, human--LLM).
All data and metrics are exportable as JSON or CSV.

\section{Application and Preliminary Evaluation}
\label{sec:counseling}

LLARS is actively used in the development of CAIA~\cite{steigerwald2025caia}, an AI-assisted system for text-based psychosocial counselling where LLM prompts transform unstructured conversations into structured outputs such as subject lines, case summaries, or response suggestions.
Counselling researchers and developers collaboratively refined prompts for seven functionalities using LLARS, and the platform also motivated two domain-specific evaluation types (Figure~\ref{fig:scenario-wizard}): \emph{Mail Rating} and \emph{Authenticity}.
For automated subject line generation~\cite{steigerwald2025subjectline}, the full pipeline was exercised end-to-end: batch generation produced 253~subject lines across 11~LLMs, and five professionals together with an LLM evaluator rated them in LLARS (1,518~assessments), directly informing model selection.

Semi-structured interviews with six domain experts (counselling researchers and practitioners) and three developers confirmed that the central benefit is bridging the gap between disciplines: the collaborative editor eliminated the ``translation work'' of formulating requirements in documents and e-mail, letting domain experts directly co-author prompts without technical details.
Developers valued the shared access point with version control; all domain experts rated the evaluation module as intuitive and a clear improvement over their previous Excel-based workflow.

\section{Limitations, Demo, and Future Work}

LLARS has so far been deployed in a single domain with a small user base; broader studies are needed to assess generalisability, and the current evaluation relies on qualitative interviews rather than a controlled study.
Our live demo walks attendees through the full pipeline, from collaborative prompt editing through batch generation to evaluation with live metrics.
