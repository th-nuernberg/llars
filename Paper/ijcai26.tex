\begin{abstract}
We demonstrate LLARS, an open-source platform that bridges the gap between domain experts and developers for building LLM-based assistants.
LLARS integrates three modules into an end-to-end pipeline:
\textbf{Collaborative Prompt Engineering} for real-time co-authoring with version control and instant LLM testing,
\textbf{Batch Generation} for systematic output production across prompts\,$\times$\,models\,$\times$\,data with cost control, and
\textbf{Evaluation} for human and LLM assessors to rate, rank, label, or compare outputs with live inter-rater agreement metrics.
Interviews with six domain experts and three developers in psychosocial counselling confirmed that LLARS eliminates the ``translation work'' between disciplines and enables domain experts to directly co-author prompts and conduct evaluations independently.

\section{Introduction}

Organizations building LLM-based assistants for sensitive domains~\cite{chen2024critical,ling2025domain}---psychosocial counseling~\cite{stade2024behavioral}, legal analysis~\cite{katz2024bar}, medical documentation~\cite{vanveen2024clinical}---must write effective prompts, select suitable models, and rigorously evaluate outputs, requiring both domain expertise (understanding \emph{what} constitutes quality) and technical skill (structuring prompts and configuring LLM pipelines)~\cite{zamfirescu2023johnny,schulhoff2024prompt}.
Yet these two groups typically work in isolation: prompts are iterated in shared documents without version control, and no single tool supports the full cycle from collaborative prompt development through systematic evaluation.

We present LLARS (LLM Assisted Research System), an open-source platform that closes this gap with three integrated modules (Figure~\ref{fig:pipeline}).
\emph{Collaborative Prompt Engineering} lets domain experts and developers co-author prompts in a shared real-time editor with version control and instant LLM testing.
\emph{Batch Generation} systematically produces the Cartesian product of prompts\,$\times$\,models\,$\times$\,data items with cost estimation and budget control.
\emph{Evaluation} lets human and LLM evaluators rate, rank, label, or compare outputs with live agreement metrics.
Each module supports JSON/CSV export and connects to any OpenAI-compatible provider.

Table~\ref{tab:comparison} surveys twelve existing platforms along these dimensions.
Agenta~\cite{agenta2024} combines prompt management with LLM-as-judge scoring but lacks real-time co-editing and structured human evaluation with agreement metrics.
Phoenix~\cite{phoenix2024}, Langfuse~\cite{langfuse2024}, and W\&B~Weave~\cite{weave2024} provide dataset experiments and per-output feedback but focus on tracing rather than systematic batch generation or multi-evaluator campaigns.
ChainForge~\cite{arawjo2024chainforge} supports visual multi-model comparison but omits human evaluation.
Label~Studio~\cite{labelstudio2024} and Argilla~\cite{argilla2024} offer structured human assessment---Label~Studio computes Krippendorff's~$\alpha$, Argilla provides agreement analytics via its API---but require externally generated outputs, disconnecting evaluation from prompt development.
LangSmith~\cite{langsmith2024}, Braintrust~\cite{braintrust2024}, Maxim~\cite{maxim2024}, and Vellum~\cite{vellum2024} link prompting to evaluation but are closed-source, lack collaborative editing, and omit batch generation as a pipeline stage.
LLARS is an open-source platform to unify collaborative prompt engineering, Cartesian-product batch generation, and hybrid human--LLM evaluation with agreement analytics, enabling developers and domain experts to work together from prompt development through rigorous assessment.


\section{Platform Overview}

LLARS runs as a containerized web application connecting to any OpenAI-compatible LLM provider, whether commercial or self-hosted.
The platform emerged from close collaboration with counselling researchers and practitioners: each module was shaped by recurring usability sessions and revised across multiple development iterations to match real workflow needs.
Teams register providers with their own API keys; models then appear throughout the platform.

\subsection{Collaborative Prompt Engineering}

The prompt engineering module (Figure~\ref{fig:prompt-editor}) lets multiple users edit the same prompt simultaneously in a real-time collaborative editor built on Yjs~\cite{jahns2023yjs}.
Prompts are structured as reorderable message blocks (system instructions, few-shot examples, user templates) with \texttt{\{\{variable\}\}} placeholders that the platform extracts into a shared palette for the team.
While editing continues, any collaborator can test the current prompt against a selected LLM: the response streams back instantly, and multiple team members can test different models or sample data in parallel.
A version control system tracks every saved revision with multi-author attribution, supporting comparison and rollback.
Finalized prompts are exported as JSON or passed directly to batch generation.

\subsection{Batch Generation}

Once prompts are refined, the batch generation module scales them to full datasets.
Consider a research team that has collaboratively developed two candidate prompts---one producing structured bullet-point analyses, the other generating narrative summaries---and wants to compare them across a self-hosted and a commercial model.
A wizard guides users through three data source modes, each suited to a different stage of development.

In \emph{new-data mode}, users upload a CSV or JSON file whose columns map to the prompt's template variables.
The team uploads 50~counselling email threads as JSON, each entry containing \texttt{subject} and \texttt{content} fields matching the variables defined during prompt engineering.
LLARS computes the full Cartesian product: 50~threads\,$\times$\,2~prompts\,$\times$\,2~models\,=\,200~outputs.
Before execution, a generation matrix previews this combination and estimates cost based on model pricing and projected token counts; optional budget caps pause jobs that exceed a threshold.

If the team later adds a third model, \emph{previous-generation mode} lets them re-run the same 50~threads with the new model---producing 100~additional outputs that join the existing provenance chain without re-uploading data.
This is also useful when the team refines a prompt after reviewing initial outputs: the revised prompt can be applied to the original dataset in a single step.

\emph{Prompt-only mode} serves a different purpose.
Before investing in data preparation, teams send multiple prompts to multiple models without any external data---for instance, testing a structured versus a narrative system prompt across three candidate models to compare output style and quality, yielding six responses to review rather than hundreds.
This lightweight comparison helps narrow the search space before committing to a full data-intensive run.

Results stream in real time, and every output records its full provenance---source item, prompt version, model, parameters, token count, and cost---exportable as CSV or JSON.
This provenance is central to the pipeline: when outputs reach evaluation, assessors see only blinded labels, and the platform maps their judgments back to specific model--prompt combinations, directly informing selection.
Completed batches transfer into evaluation with a single click, preserving all attribution.

\subsection{Evaluation}

The evaluation module brings human and LLM evaluators together in a single framework.
Four evaluation types are available (Figure~\ref{fig:scenario-wizard}): \emph{rating} with configurable multi-dimensional Likert scales, \emph{ranking} into ordered buckets, \emph{labeling} for binary or multi-class classification, and pairwise \emph{comparison}; the type system is extensible (Section~\ref{sec:counseling}).
An AI-assisted Scenario Wizard analyzes uploaded data to recommend the most suitable type; scenarios can also be created directly from batch generation outputs, preserving model attribution.

Scenario owners control how items are distributed---all items to every evaluator for full inter-rater analysis, or configurable subsets for faster throughput.
Where most tools limit LLMs to pairwise judging~\cite{zheng2024judging}, LLARS treats them as full evaluation participants that rate, rank, label, and compare alongside humans using customizable type-specific prompts.
Agreement metrics update live---Krippendorff's~$\alpha$, Cohen's and Fleiss'~$\kappa$, Kendall's~$\tau$, and ICC---filterable by evaluator pairing (human--human, LLM--LLM, human--LLM).
All data, metrics, and configurations are exportable as JSON or CSV.

\section{Deployment and User Study}
\label{sec:counseling}

LLARS is actively used in several research projects in psychosocial counselling and beyond.
The Virtual Client~\cite{rudolph2024virtualclient} leverages LLARS to develop and evaluate prompts that power generative-AI-based simulations of client interactions for counsellor training.
CAIA~\cite{steigerwald2025caia} is a co-designed AI assistant for text-based online counselling that augments professional practice through seven LLM-driven functionalities---extracting, summarising, and interpreting client communications alongside the traditional email exchange.
One of CAIA's functionalities, automated subject line generation, served as a full end-to-end case study~\cite{steigerwald2025subjectline}: prompts were collaboratively refined in LLARS, batch generation produced 253~subject lines across 11~LLMs of varying size, and five professionals together with an LLM evaluator rated them (1,518~assessments)---determining how small a model could be while still producing meaningful results.
LLARS also motivated two domain-specific evaluation types (Figure~\ref{fig:scenario-wizard}): \emph{Mail Rating} and \emph{Authenticity}.

Semi-structured interviews with six domain experts and three developers confirmed that consolidating the full pipeline in a single platform saves considerable time.
Regarding \emph{prompt engineering}, domain experts found that the collaborative editor made prompt authoring straightforward---particularly when working with self-hosted models, where no external tooling or playground exists.
For \emph{batch generation}, developers valued that prompts, data, and outputs remain in one place, eliminating the manual export--import cycles between separate tools that previously slowed iteration.
The \emph{evaluation} module was rated as immediately understandable by all domain experts: clear progress indicators show which items have been assessed and which remain, letting evaluators resume across sessions without losing context.
Across all three modules, both groups emphasised that co-locating the entire workflow removed the ``translation work'' between disciplines and created the experience of ``truly working together on the same task.''

\section{Limitations, Demo, and Future Work}

LLARS has so far been deployed in a single domain with a small user base, so broader studies are needed to assess generalisability.
The quality of LLM evaluator judgments depends on careful prompt design, and the current evaluation relies on qualitative interviews rather than a controlled study.
While the variable system helps prevent overfitting during prompt development, practitioners noted that prompts optimised on test cases can still underperform on unseen real-world data---a limitation inherent to any iterative prompt engineering process.

Our live demo walks attendees through the full pipeline, from collaborative prompt editing through batch generation to evaluation with live metrics.
Future work targets a controlled user study comparing LLARS against baseline workflows (shared documents and spreadsheets) and cross-domain validation in legal and medical settings.
